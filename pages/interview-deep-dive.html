1 DNS Sinkhole

üí¨ What to Say in Interview

"A DNS sinkhole redirects malicious domain lookups to a controlled IP instead of the real attacker server. It breaks the malware's communication channel and helps us detect infected machines."

üß† Understand This (The Real Explanation)

How DNS Normally Works:

When you type "google.com", your computer asks a DNS server "what's the IP address?" DNS responds with "142.250.x.x" and your browser connects there.

How Malware Uses DNS:

Malware needs to "phone home" to get instructions from the attacker. It's programmed with a domain like "evil-c2-server.com". When it runs, it does a DNS lookup to find the attacker's server IP, then connects to receive commands (download more malware, steal data, encrypt files, etc.)

What a Sinkhole Does:

We configure our DNS servers to LIE about malicious domains. When malware asks "where is evil-c2-server.com?", instead of returning the real attacker IP, we return an IP we control (like 10.0.0.100 - an internal server or black hole). The malware connects to our server instead of the attacker's. It can't get instructions, so the attack chain breaks.

Visual: Normal vs Sinkholed

NORMAL (Bad):
Malware ‚Üí "Where is evil-c2.com?" ‚Üí DNS ‚Üí "It's 185.x.x.x (attacker)"
Malware ‚Üí Connects to 185.x.x.x ‚Üí Gets instructions ‚Üí Does bad things

SINKHOLED (Good):
Malware ‚Üí "Where is evil-c2.com?" ‚Üí DNS Sinkhole ‚Üí "It's 10.0.0.100 (us)"
Malware ‚Üí Connects to 10.0.0.100 ‚Üí Gets nothing ‚Üí Attack fails
Security Team ‚Üí Sees connection to sinkhole ‚Üí Knows machine is infected!

Why It's Powerful:

  • ‚Ä¢ Breaks attacks - Malware can't communicate with attacker
  • ‚Ä¢ Detection - Any machine trying to reach sinkhole IP = infected
  • ‚Ä¢ Scale - Block thousands of malicious domains with one config
  • ‚Ä¢ No endpoint agent needed - Works at network level

üåç Real World Example

Scenario: You get threat intel that "malware-update.xyz" is a C2 domain used by ransomware.

Action: Add it to your DNS sinkhole. Now if ANY machine in your network tries to reach it, they get your sinkhole IP instead.

Result: Next morning, you see 3 machines tried to connect to the sinkhole IP. You just found 3 infected machines BEFORE the ransomware could activate!

üéØ Possible Follow-up Questions

Q: How do you get the list of malicious domains?

A: Threat intelligence feeds (commercial like Recorded Future, free like abuse.ch), industry sharing groups (ISACs), your own incident investigations, and vendor feeds built into NGFWs.

Q: Can attackers bypass this?

A: Yes - they can hardcode IPs instead of domains, use DNS-over-HTTPS to bypass your DNS, or use domain generation algorithms (DGAs) that create random domains. That's why sinkholing is one layer, not the only defense.

Q: Where do you implement it?

A: Internal DNS servers (Windows DNS, BIND), NGFWs with DNS filtering (Palo Alto, Fortinet), cloud DNS security (Cisco Umbrella, Zscaler), or even Pi-hole for small environments.

2 NGFW vs Traditional Firewall

üí¨ What to Say

"Traditional firewalls work at Layer 3-4 - they see IPs and ports. NGFWs work at Layer 7 - they understand applications. So even if two apps use port 443, the NGFW knows the difference and can apply different policies."

üß† Understand This

OSI Model Quick Refresher:

Layer 7 - Application   ‚Üê NGFW sees HERE (HTTP, HTTPS, DNS, apps)
Layer 6 - Presentation
Layer 5 - Session
Layer 4 - Transport     ‚Üê Traditional FW sees HERE (TCP/UDP ports)
Layer 3 - Network       ‚Üê Traditional FW sees HERE (IP addresses)
Layer 2 - Data Link
Layer 1 - Physical

Traditional Firewall Limitation:

A traditional firewall rule says "Allow TCP port 443 from 10.0.0.0/24 to any". It doesn't know WHAT'S on port 443. Could be:

  • ‚Ä¢ Legitimate HTTPS to your bank
  • ‚Ä¢ Someone tunneling SSH over 443 to bypass controls
  • ‚Ä¢ Malware C2 traffic pretending to be HTTPS
  • ‚Ä¢ Dropbox file uploads (data exfiltration risk)

Traditional FW allows ALL of these because port 443 is allowed. It's blind to the application.

How NGFW Sees Differently:

NGFW does Deep Packet Inspection (DPI). It looks INSIDE the traffic to identify the actual application. Even on port 443, it can tell:

  • ‚Ä¢ "This is Salesforce" ‚Üí Allow
  • ‚Ä¢ "This is Dropbox" ‚Üí Block (or allow with logging)
  • ‚Ä¢ "This is SSH tunneled over HTTPS" ‚Üí Block
  • ‚Ä¢ "This matches malware signature" ‚Üí Block + Alert

Key NGFW Features:

Application Awareness (App-ID)

Identifies 3000+ applications regardless of port, encryption, or evasion technique

User Awareness (User-ID)

Rules based on AD users/groups, not just IP addresses. "Allow Sales team to use Salesforce"

SSL/TLS Decryption

Decrypts HTTPS to inspect traffic, re-encrypts before sending. Can see inside encrypted connections.

Integrated IPS

Intrusion Prevention built-in. Blocks exploits, not just at perimeter but inline.

Threat Intelligence

Real-time updates on malicious IPs, domains, file hashes. Auto-blocks known bad.

Sandboxing

Unknown files sent to cloud sandbox for analysis before allowing.

üåç Real World Example

Problem: Employee wants to use personal Dropbox to "work from home easier"

Traditional FW: Can't stop it - Dropbox uses HTTPS (443), which is allowed.

NGFW: Rule says "Block Dropbox, Allow OneDrive". Even though both use 443, NGFW knows the difference. Employee sees "This application is blocked by company policy."

üéØ Follow-up Questions

Q: What about performance impact of DPI?

A: Yes, there's overhead. That's why you size NGFWs based on throughput WITH features enabled, not just raw throughput. A firewall rated at 10Gbps might only do 2Gbps with full DPI, SSL decrypt, and threat prevention.

Q: What NGFW vendors have you used?

A: Palo Alto (most common in enterprise), Fortinet FortiGate (good price/performance), Cisco Firepower, Check Point. Each has strengths - Palo Alto's App-ID is industry-leading, Fortinet has great price for SMB.

Three Core Principles (VLA):

1. Verify Explicitly

Authenticate and authorize based on ALL available data points:

  • ‚Ä¢ User identity (who are you?)
  • ‚Ä¢ Device health (is your laptop compliant? patched? encrypted?)
  • ‚Ä¢ Location (are you in expected country?)
  • ‚Ä¢ Network (corporate WiFi vs coffee shop?)
  • ‚Ä¢ What you're accessing (is this normal for your role?)
  • ‚Ä¢ Risk signals (impossible travel? new device?)

2. Least Privilege Access

Give minimum access needed, for minimum time:

  • ‚Ä¢ Just-In-Time (JIT) access - request when needed, expires after
  • ‚Ä¢ Just-Enough-Access (JEA) - only what you need for task
  • ‚Ä¢ No standing admin rights - elevate only when required

3. Assume Breach

Design as if attackers are ALREADY inside:

  • ‚Ä¢ Micro-segmentation - every workload isolated
  • ‚Ä¢ Encrypt everything - even internal traffic
  • ‚Ä¢ Continuous monitoring - detect lateral movement
  • ‚Ä¢ Minimize blast radius - breach of one thing doesn't = breach of everything

Zero Trust in Practice (Microsoft Example):

User tries to access SharePoint:

  1. 1. Entra ID verifies identity (password + MFA)
  2. 2. Conditional Access checks: Is device compliant? Location OK? Sign-in risk low?
  3. 3. If all pass ‚Üí Grant access to JUST SharePoint (not everything)
  4. 4. If risky ‚Üí Require additional verification or block
  5. 5. Session controls - can't download to unmanaged device
  6. 6. Continuous evaluation - if device becomes non-compliant mid-session, access revoked

üåç Real World Example

Old way: User connects to VPN ‚Üí Full network access ‚Üí Can reach any server

Zero Trust: User authenticates ‚Üí Conditional Access checks device ‚Üí Only granted access to specific apps they need ‚Üí Each app verifies again ‚Üí Session monitored for anomalies

Result: Even if attacker steals credentials, they can't access anything because they fail device compliance check. Even if they compromise one app, they can't move laterally because everything else requires separate verification.

4 Privileged Access Management (PAM)

üí¨ What to Say

"PAM controls access to high-privilege accounts like domain admins and root. Key capabilities are: credential vaulting so nobody knows the actual password, session recording for audit, and just-in-time access so privileges are temporary. Most breaches involve privileged credentials, so protecting them is critical."

üß† Understand This

Why Privileged Accounts Are Dangerous:

Think about what these accounts can do:

  • ‚Ä¢ Domain Admin - Control entire Active Directory, every user, every computer
  • ‚Ä¢ Root/Local Admin - Full control of a system, can install anything
  • ‚Ä¢ Database Admin - Access all data, can export entire databases
  • ‚Ä¢ Cloud Admin - Create/delete resources, access all storage

If attacker gets ONE of these, game over. That's why they're the #1 target.

The Problem PAM Solves:

WITHOUT PAM:
- Admin knows password "SuperSecret123"
- Password never changes
- Admin uses it from any machine
- No record of what admin did
- If admin leaves, password still works
- If attacker steals password, works forever

WITH PAM:
- Password stored in encrypted vault
- Admin doesn't know actual password
- Password rotates automatically (daily/hourly)
- Admin "checks out" access when needed
- Every action recorded on video
- Access expires after 4 hours
- Attacker steals hash? It's already rotated!

Core PAM Capabilities:

1. Credential Vaulting

Passwords stored encrypted in secure vault. Users never see actual password - they "check out" access through the PAM system which injects credentials automatically.

2. Automatic Rotation

Passwords changed automatically on schedule. Even if attacker dumps hashes from memory, they're useless tomorrow.

3. Session Recording

Video recording of everything done during privileged session. For auditing ("what did the contractor do?") and forensics ("how did the breach happen?").

4. Just-In-Time (JIT) Access

No standing privileges. Request access ‚Üí Approval workflow ‚Üí Get access for limited time ‚Üí Auto-revoke. You're only admin when you need to be.

5. Privileged Access Workstations (PAW)

Admins can only use privileged credentials from hardened, dedicated machines. Can't accidentally use domain admin from your regular browsing laptop.

Common PAM Tools:

  • ‚Ä¢ CyberArk - Enterprise leader, full featured
  • ‚Ä¢ BeyondTrust - Strong in endpoint privilege management
  • ‚Ä¢ Azure PIM (Privileged Identity Management) - Cloud-native for Azure/M365 roles
  • ‚Ä¢ HashiCorp Vault - Popular for DevOps/secrets management

üåç Real World: How I'd Explain Azure PIM

"In Azure PIM, I'm 'eligible' for Global Admin but don't have it active. When I need to make a change:

  1. 1. I request activation in the portal
  2. 2. I provide justification ('need to modify Conditional Access policy')
  3. 3. I complete MFA
  4. 4. I get Global Admin for 8 hours max
  5. 5. Everything I do is logged
  6. 6. Access auto-revokes after 8 hours

So 95% of the time I have no admin rights at all - much smaller attack surface."

5 DLP Implementation (Your Real Experience!)

üí¨ What to Say

"I led a DLP migration from Symantec to Microsoft Purview. The key was phased rollout: Discovery first to understand data, then Audit mode to see what policies would catch, Tune to reduce false positives, Warn mode to educate users, and finally Block for high-risk actions. Took about 6 months but we reduced false positives by 40% compared to the old system."

üß† Understand This

What DLP Actually Does:

Data Loss Prevention monitors data movement and blocks/alerts when sensitive data goes somewhere it shouldn't:

  • ‚Ä¢ Emailing credit card numbers to personal Gmail
  • ‚Ä¢ Uploading source code to public GitHub
  • ‚Ä¢ Copying customer PII to USB drive
  • ‚Ä¢ Printing HR documents with SSNs

Why Phased Rollout is Critical:

WRONG WAY (What NOT to do):
Day 1: Turn on blocking ‚Üí Finance can't send reports
       ‚Üí Sales can't email contracts ‚Üí Executives angry
       ‚Üí IT flooded with tickets ‚Üí DLP turned off
       ‚Üí Project labeled "failed"

RIGHT WAY (Phased):
Month 1-2: AUDIT MODE - Just watch, don't block anything
           ‚Üí Learn what sensitive data exists
           ‚Üí See normal business workflows
           ‚Üí Identify false positive patterns

Month 3-4: TUNE - Adjust detection rules
           ‚Üí Add exclusions for legitimate workflows
           ‚Üí Refine sensitivity labels
           ‚Üí Reduce noise

Month 5:   WARN MODE - Notify users but allow override
           ‚Üí User sees "This looks sensitive, are you sure?"
           ‚Üí User can proceed with justification
           ‚Üí Educates users about data handling
           ‚Üí Gathers data on overrides

Month 6+:  BLOCK MODE (graduated)
           ‚Üí Block highest risk first (CC numbers external)
           ‚Üí Warn for medium risk
           ‚Üí Monitor for low risk

Your Purview Migration Story (Use This!):

Challenge: Migrating from Symantec DLP (legacy) to Microsoft Purview Endpoint DLP while maintaining protection.

Approach:

  1. 1. Ran both systems in parallel during transition
  2. 2. Rebuilt detection logic - Symantec regex ‚â† Microsoft sensitive info types
  3. 3. Used Purview's built-in trainable classifiers (ML-based detection)
  4. 4. Phased: Audit ‚Üí Warn ‚Üí Block over 6 months
  5. 5. Created custom sensitive info types for company-specific data

Result: 40% fewer false positives because of better tuning. Full integration with Microsoft ecosystem (Defender, Sentinel alerts).

üéØ Follow-up Questions

Q: How do you handle false positives?

A: Create exclusions for known good workflows (e.g., finance reports to auditors), use contextual rules (SSN pattern in HR email = OK, SSN to personal email = block), regularly review and tune based on user feedback.

Q: What data did you protect?

A: PII (SSN, credit cards, passport numbers), financial data, intellectual property (source code, designs), health information. Prioritized based on regulatory requirements and business risk.

Q: How did users react?

A: Initial pushback, but warn mode helped - users understood what was sensitive. Made sure to communicate WHY before rollout. Provided training. The key is not surprising people with blocks.

6 OT vs IT Security

üí¨ What to Say

"The biggest difference is priorities: IT is about data confidentiality, OT is about availability and safety. In IT, we patch monthly. In OT, some systems can't be patched for years because of vendor certification, downtime costs, or legacy OS. You can't just run vulnerability scans against OT - it might crash a PLC. The approach is heavy segmentation, passive monitoring, and compensating controls."

üß† Understand This

What is OT (Operational Technology)?

Hardware and software that monitors/controls physical processes:

  • ‚Ä¢ SCADA - Supervisory Control and Data Acquisition (monitors/controls industrial processes)
  • ‚Ä¢ PLC - Programmable Logic Controller (controls machinery)
  • ‚Ä¢ DCS - Distributed Control System (process control)
  • ‚Ä¢ HMI - Human Machine Interface (operator screens)
  • ‚Ä¢ Manufacturing robots, power grid controls, water treatment, etc.

Key Differences Table:

Aspect IT OT
Priority Confidentiality first Availability & Safety first
Downtime tolerance Minutes to hours OK Seconds can cost millions
Patching Monthly (Patch Tuesday) Yearly or never
System age 3-5 year refresh 15-30 years
OS Current Windows/Linux Windows XP, proprietary RTOS
Scanning Active scanning OK Passive only (active can crash systems)
Breach impact Data stolen, financial loss Physical damage, safety risk, environmental

OT Security Approach:

  • ‚úì Network segmentation - Isolate OT from IT, segment within OT
  • ‚úì Passive monitoring - Tools like Claroty, Nozomi, Dragos that watch traffic without sending packets
  • ‚úì Whitelist approach - Only allow known-good traffic patterns
  • ‚úì Physical security - Lock down USB ports, console access
  • ‚úì Compensating controls - If can't patch, add more monitoring, segment more
  • ‚úó DON'T run active vulnerability scans
  • ‚úó DON'T install agents without vendor approval
  • ‚úó DON'T push patches without extensive testing

üåç Why This Matters (Real Incidents)

  • ‚Ä¢ Colonial Pipeline (2021) - Ransomware on IT side, but shut down OT pipelines as precaution. East coast fuel shortage.
  • ‚Ä¢ Ukraine Power Grid (2015) - Attackers accessed SCADA, remotely opened breakers. 230,000 people lost power.
  • ‚Ä¢ Triton/TRISIS (2017) - Malware targeted SAFETY systems in petrochemical plant. Could have caused explosion.

This is why OT security is critical - it's not just data, it's physical safety.

  • ‚úì Baseline existing issues - Don't fail on day 1 for 1000 old issues
  • ‚úì Fast feedback - SAST results in minutes, not hours
  • ‚úì Developer-friendly tools - IDE integration, PR comments, not just reports
  • ‚úó Avoid becoming the "team that blocks everything"
