1. DLP Policy Lifecycle (Audit → Warn → Block) 

┌─────────────────────────────────────────────────────────────────────────────────────┐
│                          DLP POLICY LIFECYCLE                                       │
│                    (The Golden Rule: Never start with Block)                        │
└─────────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────────┐
│ PHASE 1: AUDIT MODE (2-4 weeks)                                                     │
│ "See everything, block nothing"                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ POLICY SETTINGS:                                                                │
    │ • Action: Audit (log only, no user impact)                                     │
    │ • Notifications: None to users                                                 │
    │ • Alerts: Security team only                                                   │
    │                                                                                 │
    │ WHAT YOU'RE LEARNING:                                                          │
    │ • What data is actually being shared?                                          │
    │ • Which users trigger the most alerts?                                         │
    │ • Are there legitimate business workflows we didn't know about?                │
    │ • What's the false positive rate?                                              │
    │                                                                                 │
    │ WEEKLY REVIEW:                                                                  │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ Alert Volume: 1,245 this week                                               ││
    │ │ Top Triggers:                                                                ││
    │ │ • Credit Card Numbers: 520 (42%)                                            ││
    │ │ • SIN/SSN: 312 (25%)                                                        ││
    │ │ • "Confidential" keyword: 413 (33%)                                         ││
    │ │                                                                              ││
    │ │ Analysis: 60% appear legitimate business (HR, Finance)                      ││
    │ │ Action: Need to tune or add exceptions                                      ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    └─────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ (When FP rate < 20%)
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ PHASE 2: WARN/NOTIFY MODE (2-4 weeks)                                               │
│ "Educate users, still don't block"                                                  │
└─────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ POLICY SETTINGS:                                                                │
    │ • Action: Warn (user sees notification, can override)                          │
    │ • Notifications: Policy tip shown to user                                      │
    │ • Override: Allowed with business justification                                │
    │                                                                                 │
    │ USER EXPERIENCE:                                                               │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ ⚠️ This file contains sensitive information (Credit Card Numbers)           ││
    │ │                                                                              ││
    │ │ Sharing this externally may violate company policy.                         ││
    │ │                                                                              ││
    │ │ [Override - I have business need] [Cancel]                                  ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    │                                                                                 │
    │ WHAT YOU'RE LEARNING:                                                          │
    │ • Do users understand why they're being warned?                                │
    │ • What percentage override vs cancel?                                          │
    │ • Are the override justifications legitimate?                                  │
    │ • Do we need more user training?                                               │
    │                                                                                 │
    │ SUCCESS CRITERIA:                                                              │
    │ • Override rate stabilizes (users learn)                                       │
    │ • No major business process complaints                                         │
    │ • False positive rate < 10%                                                    │
    └─────────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ (When FP rate < 10% and stable)
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ PHASE 3: BLOCK MODE (Enforcement)                                                   │
│ "Now we protect, but with escape valves"                                            │
└─────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ POLICY SETTINGS:                                                                │
    │ • Action: Block (with override for some policies)                              │
    │ • Notifications: User notified of block                                        │
    │ • Override: Manager approval for some, no override for others                  │
    │                                                                                 │
    │ TIERED APPROACH:                                                               │
    │                                                                                 │
    │ HIGH SENSITIVITY (No override):                                                │
    │ • PCI data to unauthorized external recipients                                 │
    │ • Healthcare records (PHI) to personal email                                   │
    │ • Source code to competitors                                                   │
    │                                                                                 │
    │ MEDIUM SENSITIVITY (Manager override):                                         │
    │ • Internal-only documents shared externally                                    │
    │ • Customer lists to non-approved vendors                                       │
    │                                                                                 │
    │ LOW SENSITIVITY (User override with justification):                            │
    │ • "Confidential" marked documents                                              │
    │ • Internal project information                                                 │
    └─────────────────────────────────────────────────────────────────────────────────┘

3. DLP Classifier Layering Strategy 

┌─────────────────────────────────────────────────────────────────────────────────────┐
│                         DLP CLASSIFIER LAYERING                                     │
│                  (Maximize detection, minimize false positives)                     │
└─────────────────────────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ LAYER 1: BUILT-IN SENSITIVE INFORMATION TYPES (SITs)                            │
    │                                                                                 │
    │ Use for: Standard patterns (credit cards, SSN, passport numbers)               │
    │                                                                                 │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ Examples:                                                                    ││
    │ │ • Credit Card Number (Luhn checksum validated)                              ││
    │ │ • U.S. Social Security Number                                               ││
    │ │ • Canada Social Insurance Number                                            ││
    │ │ • Passport Numbers (various countries)                                      ││
    │ │ • Bank Account Numbers                                                       ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    │                                                                                 │
    │ Pros: Ready to use, Microsoft maintained, checksum validated                   │
    │ Cons: Generic patterns, may need context tuning                                │
    └─────────────────────────────────────────────────────────────────────────────────┘
                              +
    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ LAYER 2: CUSTOM SENSITIVE INFORMATION TYPES                                     │
    │                                                                                 │
    │ Use for: Company-specific patterns                                             │
    │                                                                                 │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ Examples:                                                                    ││
    │ │ • Internal project codes: "PRJ-[A-Z]{3}-[0-9]{4}"                           ││
    │ │ • Customer account numbers: "CUST[0-9]{8}"                                  ││
    │ │ • Employee IDs: "EMP-[0-9]{6}"                                              ││
    │ │ • Product formulas: "FORMULA-[A-Z]{2}[0-9]{3}"                              ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    │                                                                                 │
    │ Pros: Tailored to your organization, reduces false positives                   │
    │ Cons: Requires creation and maintenance                                        │
    └─────────────────────────────────────────────────────────────────────────────────┘
                              +
    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ LAYER 3: EXACT DATA MATCH (EDM)                                                 │
    │                                                                                 │
    │ Use for: Specific known values (actual customer names, account numbers)        │
    │                                                                                 │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ How it works:                                                                ││
    │ │ 1. Upload database of actual sensitive values (hashed)                      ││
    │ │ 2. DLP matches exact values, not patterns                                   ││
    │ │                                                                              ││
    │ │ Example: Upload customer database                                           ││
    │ │ • "John Smith" + "123-45-6789" + "john.smith@email.com"                     ││
    │ │ • Only triggers when actual customer data is found                          ││
    │ │ • Not random SSN patterns                                                    ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    │                                                                                 │
    │ Pros: Lowest false positive rate, detects actual sensitive data               │
    │ Cons: Requires data upload and refresh, more complex setup                     │
    └─────────────────────────────────────────────────────────────────────────────────┘
                              +
    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ LAYER 4: KEYWORD CONTEXT                                                        │
    │                                                                                 │
    │ Use for: Increasing confidence of detection                                    │
    │                                                                                 │
    │ ┌─────────────────────────────────────────────────────────────────────────────┐│
    │ │ Require pattern + context keywords:                                         ││
    │ │                                                                              ││
    │ │ Credit Card + ("payment", "billing", "card number", "CVV")                  ││
    │ │ SSN + ("social security", "SSN", "tax ID", "employee")                      ││
    │ │                                                                              ││
    │ │ This filters out:                                                           ││
    │ │ • Random 16-digit numbers that aren't credit cards                          ││
    │ │ • Phone numbers that match SSN pattern                                      ││
    │ └─────────────────────────────────────────────────────────────────────────────┘│
    │                                                                                 │
    │ Pros: Significantly reduces false positives                                    │
    │ Cons: May miss data without context (but usually want context anyway)          │
    └─────────────────────────────────────────────────────────────────────────────────┘
                              =
    ┌─────────────────────────────────────────────────────────────────────────────────┐
    │ COMBINED: HIGH-CONFIDENCE DETECTION                                             │
    │                                                                                 │
    │ Example Policy: "Protect Customer PII"                                         │
    │                                                                                 │
    │ DETECT when ANY of:                                                            │
    │ ├── Credit Card (built-in) + payment keywords                                  │
    │ ├── SSN (built-in) + employee/customer keywords                                │
    │ ├── Customer Account Number (custom SIT)                                       │
    │ └── EDM match against customer database                                        │
    │                                                                                 │
    │ CONFIDENCE: High (85%+) → Block                                                │
    │ CONFIDENCE: Medium (65-84%) → Warn                                             │
    │ CONFIDENCE: Low (<65%) → Audit only                                            │
    └─────────────────────────────────────────────────────────────────────────────────┘
