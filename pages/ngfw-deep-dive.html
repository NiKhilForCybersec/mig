Security Zones (Common Setup)

  • Untrust / Outside: Internet-facing, highest risk
  • DMZ: Public-facing servers (web, mail, DNS)
  • Guest: Isolated visitor/BYOD network
  • Trust / Inside: Internal users, endpoints
  • Server / Datacenter: Critical servers, databases
  • Management: Admin access, out-of-band

High Availability Modes

  • Active/Passive: One firewall handles traffic, other on standby. Simpler, no session sharing needed.
  • Active/Active: Both handle traffic, load balanced. Better performance, more complex.
  • HA Heartbeat: Dedicated link between firewalls to detect failure (typically via HA ports).
  • Session Sync: Active shares session table with passive for seamless failover.

6 Interview Questions & Answers

Q: What's the difference between a stateful firewall and NGFW?

"Stateful firewalls track connection state and work at layers 3-4 - they see IP addresses and ports. They allow return traffic for established connections but can't see what's inside the packets. NGFWs add layer 7 visibility - they identify applications regardless of port, integrate user identity from AD, decrypt SSL to inspect encrypted traffic, and include IPS, URL filtering, and threat intelligence. So NGFW is stateful plus application awareness plus integrated security services."

Q: How do you troubleshoot a user reporting blocked traffic?

"First, I get specifics - source IP, destination, port, application. Then I check traffic logs filtered by that source IP around the time of the issue. If I see a deny, I identify which rule matched - implicit deny or explicit block. I also check App-ID - sometimes the app is misidentified. If the traffic should be allowed, I create a rule through proper change management. If it's being blocked for good reason, I explain to the user why and offer alternatives if possible. I always test after making changes using packet capture or session logging to confirm the fix."

Q: How do you approach firewall rule cleanup?

"I run reports showing rules with zero hits over 90 days - these are candidates for removal. But I don't just delete them - first I disable with a date tag, wait 30 days, monitor for any breakage. I also look for overlapping rules where a broader rule makes specific ones redundant. Shadow rules that never match because something above catches the traffic first. And duplicate rules that do the same thing. I document the cleanup in change management and schedule it during low-risk periods. The goal is reducing attack surface and improving performance without breaking anything."

Q: Explain how you would implement a new NGFW from scratch.

"I'd start with requirements - throughput needs, features required, HA requirements, budget. Then architecture design - where does it sit, what zones, what traffic flows through it. For the actual deployment: initial config in a lab first, configure interfaces and zones, set up management access, integrate with AD for User-ID, configure logging to SIEM. Then build the base policy - start with deny-all, explicitly allow what's needed. Enable security profiles gradually - IPS first, then URL filtering, then SSL decryption for high-risk categories. Test thoroughly before cutover, have a rollback plan. After go-live, monitor closely for a week, tune false positives, then enable additional features incrementally."

Q: What are the security considerations for SSL decryption?

"SSL decryption is essential because 90%+ of traffic is encrypted - without it you're blind. But there are considerations: Performance - decryption is CPU intensive, need to size hardware appropriately. Privacy - get HR/legal approval, create policy. Exclusions - some traffic should never be decrypted: banking, healthcare, certificate-pinned apps. Compliance - some regulations prohibit decrypting certain data types. Certificate management - your CA cert needs to be trusted by all endpoints, deploy via GPO. Start with high-risk categories first - newly registered domains, unknown categories - then expand based on visibility needs."

Q: You mentioned you've worked with [Palo Alto/Fortinet]. Walk me through a typical day.

"Morning starts with checking overnight alerts - any critical threats blocked, any HA events, any capacity warnings. Then I review the queue of firewall change requests - analyze each one for security risk, verify business justification, ensure least privilege. After changes are approved and implemented, I test them and update the ticket. Throughout the day I troubleshoot connectivity issues - users reporting blocked apps, which means checking logs, identifying the blocking rule, and either adjusting policy or explaining why it's blocked. I also spend time on proactive work - reviewing unused rules for cleanup, optimizing rule order, checking for software updates and planning patching windows. Weekly I review threat logs for patterns, monthly I generate reports for management."
